$note1
底層CPU機制
快取
L1/L2/L3 cache
L1 cpu核心內部 最快(4 cycles) 最小(32-128KB) 
L2 每個核心專屬 (12 cycles) (256KB-1MB)
L3 所有核心共用 (40 cycles) (4MB-64MB+)
RAM 主機板 (200+ cycles) (RAM size)
cpu 從 L1 往下找 
找到後 傳回 cache Line (64 bytes) 的資料
所以 int[] char[] 能夠很快的取得資料
但 Integer[] String[] 本質是 Object[] 
內部有很多標頭 所以是address[] 
雖然同樣是連續記憶體 但還要到address 才能取得值

boolean[] 因為cpu最小取得單位是 bytes(8 bits) 
所以布林值 是 1bytes 不是 1bit (避免 修改 要利用mask導致速度變慢)
可以利用專門的bitSet(1 bool 1 bit) 來節省空間

cpu遇到boolean 會用預測的方式 來運算(前面true的機率...) 
先偷跑 等待 真實value從其他cache(通常是RAM)搬回來 在做驗證
如果錯誤 從結點重跑 如果正確 結案
所以 有規律的資料 會比 沒規律的資料 要更快
如果遇到需要判斷很多次的情況：
先過濾掉最可能失敗的條件：讓 CPU 早點知道不用跑了。
減少分支：如果能用 Math.max(a, b) 取代 if (a > b)，
就用函數。因為現代 CPU 的 cmov (Conditional Move) 指令可以直接完成判斷而不產生分支，CPU 就不需要猜了。

int[n][m] 
所以2維陣列 就是 int[n*m]的 連續記憶體
因此 橫向遍歷 會比縱向遍歷要快
橫向 n, n+n.... 有規律性,且記憶體位置靠近
縱向 n, n+m.... 雖然有規律性, 但記憶體位置可能太遠了 造成
超過Cache Line的回傳 每次都要搬運新資料
或是 TLB (轉譯後備緩衝區)
虛擬記憶體：你的程式看到的地址是「虛擬」的，需要透過「頁表」轉成 RAM 的「物理」地址。
TLB：這是 CPU 內部一塊極小、極快的區域，專門紀錄最近轉換過的地址（像是一張地址地圖）。
縱向遍歷的慘劇：當你跨行跳轉時，不僅超出了 Cache Line，還可能跳出了記憶體的一「頁」(通常是 4KB)。這會導致 TLB 找不到地址，CPU 必須去 RAM 翻閱厚厚的頁表。

總結
空間局部性 (Spatial Locality)：資料在記憶體位置上靠得很近（如橫向遍歷），這讓 Cache 效率最大化。

時間局部性 (Temporal Locality)：資料在短時間內被重複使用，這讓資料能保存在 L1 不被踢出去。

$note2
進程 vs. 執行緒
進程 	:工廠(資源分配的最小單位, 自己的工廠倒了 不影響別家工廠)
執行緒 	:工人(cpu調動的最小單位, 共享工廠的東西, 可能把工廠燒了(overflow 會影響到OS的狀況 OS會主動砍掉整個工廠 避免災害擴大))
上下文切換 (搶佔式多工)

為了避免電腦卡住與看起來像多工 OS會限制所有的工人都有一段最長執行時間 (Time Slice)
時間到了(硬體定時器訊號 Timer Interrupt) OS就換人 上一個人的工具(數據)則寫入RAM 換下一個人工作
工人 A 的工具存到哪裡？ (PCB/TCB)
你問「把工具先存起來」，這在電腦裡有個專門的地址：

TCB (Thread Control Block, 執行緒控制塊)：這是 OS 在 RAM 裡為每個執行緒開的「置物櫃」。
1.程序計數器 (PC)：工人 A 讀到哪一行指令了？
2.暫存器 (Registers)：工人 A 算到一半的數字是多少？
3.棧指針 (Stack Pointer)：工人 A 目前呼叫函式的深度到哪了？

切換的 耗損
快取冷卻 (Cache Cold)
這是最致命的代價。雖然暫存器（工具）搬運很快，但 「大腦（快取）」 的記憶沒辦法搬運。
工人 A 工作時，L1/L2 快取裡全是 A 需要的數據。
工人 B 上場後，他看著快取裡的數據一臉懵逼：「這不是我要的！」
Cache Miss 狂發：工人 B 必須重新去 RAM 搬他需要的數據。
等到快取終於「熱」起來（Warm-up），工人 B 的時間片又到了，他又被叫停了...
所以執行緒太多 會造成資料不斷搬運 實際工作時間受限

java21的 虛擬執行緒
不透過OS的執行緒 由 JVM自己管理(減少記憶體開銷, 也不會讓OS頻繁調用系統)

虛擬記憶體與 Page Fault
為什麼 RAM 滿了電腦會卡死？
虛擬記憶體 (Virtual Memory)：給每個進程一個幻覺，讓它覺得自己擁有 4TB 的連續空間。
分頁 (Paging)：OS 把虛擬空間切成一小塊一小塊（4KB）。
Page Fault (缺頁中斷)：當你訪問一個地址，發現它不在 RAM 裡，而在硬碟裡。
Swap (置換)：OS 必須把 RAM 裡很久沒用的資料搬到硬碟，再把你要的搬進來。
代價：硬碟比 RAM 慢 10 萬倍。當 Page Fault 頻繁發生，CPU 就在無止盡等待硬碟搬運，這就是「卡死」的真相。

基本連線
TCP (保證順序傳輸), UDP(不保證傳輸)
SYN (Synchronize)：客戶端發起。
SYN-ACK (Synchronize-Acknowledgement)：伺服器回應。
ACK (Acknowledgement)：客戶端最後確認。
三向握手連線
SYN(A)我要連線了 -> SYN-ACK(B)OK 你有收到我的回應嗎 -> ACK(A) 有,連線正常
四次揮手斷線
A 我要斷線了(斷線 half-close狀態 只接收不做處理 避免B端 因為沒有傳輸 無法停止) -> B 好的 (傳輸剩下的資料) -> B 結束(ACK) -> A(正式CLOSE)
連線池 因為連線很浪費資源 所以預先準備好長連線池 避免每次都要握手

HTTP 歷史 (現階段 3 種都還有再使用)
HTTP/1.1 : 舊式SERVER
	頭部阻塞 HOL Blocking 
	一條窄路 單次通行的空間有限 當前方的資訊還沒傳完 後面的都會塞住
HTTP/2	 : 需要保證資料完整性的SERVER
	多工處理 將資料 切小塊(Frame) 避免阻塞
	TCP HOL Blocking 所以當其中一個封包丟失 整條路會停下來等他帶丟失的送達
HTTP/3	： 追求速度的SERVER
	非阻塞 (QUIC / UDP模式)
	UDP 特性 丟失的封包不管 
	QUIC 協定：它是 HTTP/3 的心臟。雖然底層跑的是 UDP，但 QUIC 在上面自己寫了一套「確認機制 (ACK)」。
	如果元件（封包）缺失：
	1. TCP (HTTP/2)：如果第 1 個封包掉了，第 2, 3 個封包就算到了也要等，這叫 Head-of-Line Blocking。	
	2. QUIC (HTTP/3)：如果第 1 個流（Stream）的封包掉了，只有那個流會停下來等。第 2, 3 個流如果屬於不同的檔案（例如不同的圖片），它們可以繼續傳輸，完全不受影響！
HTTP 封包
	當我們說 HTTP 封包（Packet/Message）時，它包含兩個部分：
	Header (標頭)：就像快遞單。寫著：我要去哪（URL）、我是什麼格式（JSON 還是 HTML）、我有沒有登入（Cookie）。
	Body (內容物)：這才是真正的資料。它可以是：
		HTML (網頁結構), JSON (API 資料), JPG/PNG (圖片), MP4 (影片串流)
DNS CDN 導航 跳轉
	連線 PC -> location ISP -> Root , return to PC
	DNS 電話簿 	:  經過上面的步驟 為了避免短期內重複訪問的時間 將地址存在Cache
	CDN 超商 	: 避免每次都去 遠方取得資料 先把資料放在 某個節點伺服器 減少傳輸時間 
	邊緣運算(Edge): 封包跳轉時，會經過無數個路由器。CDN 會利用 DNS 解析，把你的請求導向「物理距離最近」的節點。
	
HTTPS 加密連線(HTTP + TLS/SSL)
	2次的鑰匙交換
	pharse 1:
		非對稱加密(SERVER 持有私鑰)
		客戶端拿伺服器的公鑰，把一組隨機數字（對稱密鑰）鎖進保險箱送過去。這世界上只有持有私鑰的伺服器打得開。
	phares 2:
		Server 解鎖後 使用（對稱密鑰）
		把內容進行對稱加密 
		這樣雙方都使用 同一把鑰匙溝通
	CA憑證 :
		你怎麼知道公鑰真的是 Google 的？由 數位憑證中心 (CA) 簽名擔保，防止中間人攔截（防止有人拿假的保險箱騙你）。

加密連線
	對稱加密		: 雙方使用 同一把 鑰匙加密與解密。	效能之王。運算極快，對 CPU 負擔小，適合搬運大量數據（如 HTML, 影片）。
		DES (已淘汰)、AES (目前標準)。
	TLS 		:傳輸層安全性協定 (Transport Layer Security)。	現代標準。目前主流是 TLS 1.2 或 1.3，解決了攔截與竄改問題。
	SSL 		:TLS 的前身 (Secure Sockets Layer)。	已過時。雖然現在大家口頭還會說 SSL，但技術上指的都是 TLS。 (淘汰)
	非對稱加密	: 使用 公鑰 加密，私鑰 解密。	安全之王。運算慢（複雜數學），僅用於連線初期「交換暗號」。
		RSA (漸漸被取代)、ECC (橢圓曲線，現在主流)。
		公鑰 	: 負責關上保險箱 (加密) 在非對稱中 用來加密 但無法解密
		私鑰 	: 負責打開保險箱(解密)  負責解密

不可逆加密
	雜湊演算法 (Hashing) —— 只有去，沒有回
	代表：MD5 (已淘汰)、SHA-1 (不安全)、SHA-256 (目前標準)。
	特性：
		不可逆：你把「密碼」變成「SHA-256 亂碼」很容易，但沒人能把亂碼變回密碼。
		唯一性：只要內容改一個 bit，算出來的結果就完全不同。
		長度固定
	增加複雜度(加鹽)
		SHA-256 「唯一性」，但這有個漏洞：「123456」的 SHA-256 永遠長得一樣。
		當使用者設定密碼 123456 時，你在程式裡隨機生成一個字串 xyz。
		你存進資料庫的是：SHA256("123456" + "xyz")。
		結果：就算兩個人的密碼都是 123456，因為「鹽巴」不同，存起來的亂碼也完全不同。這讓駭客的預計算清單全部作廢。

Java 底層
JMM(Java 記憶體模型)
	1.Stack	: 執行緒的口袋 , 存基本型別(int boolean), 對象的引用(address), 跟著方法走, 方法結束就丟棄 L1/L2 Cache
	2.Heap 	: 所有執行緒共用的倉庫, 存 New對象(物件), 因此散落在各處 速度慢, 需要透過GC回收

垃圾回收(GC)
	STW(stop-the-world) 讓所有工作暫停 避免丟垃圾時 還時其他人持續丟 避免掃不完
	G1 GC(old) 		: 把Heap切小塊 哪裡垃圾多 先回收哪裡, 減少停頓
	ZGC(java 11+)	: 這是一個黑科技。它能在 CPU 繼續工作的同時進行回收，STW 時間控制在 10 毫秒 以內。
	
JIT(Just-In-Time)
	JVM 自動程式優化
	1.解釋執行 	: JVM 翻譯.class to binary line by line to cpu (慢)
	2.熱點探測	: 發現某些方法被多次呼叫 (常用代碼)
	3.機器碼編譯	: 將熱點 編譯成機械碼 (快)
	4.內聯優化	: 它甚至會把小方法(可以塞入 L1/L2 Cache的方法 通常在 35bytes以內)
				直接塞進呼叫者裡，減少跳轉開銷，這效能有時甚至能追上 C++。
					
volatile 與 CAS：高併發的守護神
	volatile：
		問題：CPU 核心 A 修改了變數，存進了自己的 L1 Cache，核心 B 卻看不到（還在看舊的 L1 資料）。
		原理：加上 volatile，OS 會強制讓核心 A 的寫入直接衝回 RAM，並讓其他核心的 L1 快取失效（MESI 協定）。這保證了 「可見性」。
	CAS (Compare And Swap)：
		原理：這是一個 CPU 硬體指令（無鎖）。它會問：「我預期這個地址的值是 5，如果是，請幫我改成 6。」
		優點：它不開鎖、不進出核心態、不產生 Context Switch。它是高併發系統中「效能最高」的同步方式。
		
逃逸分析 (Escape Analysis)
	什麼是「逃逸」？
	當你 new 一個物件時，JVM 會觀察這個物件的生命週期：
	逃逸了：這個物件被回傳給其他方法，或是賦值給全域變數（大家都能用）。
	沒逃逸：這個物件只在這個方法裡面用，方法結束它就沒用了。

	2. 逃逸分析後的三大優化
		如果 JVM 發現一個物件「沒逃逸」，它會進行以下三種瘋狂的優化：

	A. 棧上分配 (Stack Allocation) —— 幹掉 GC
		傳統邏輯：所有 new 出來的物件都在 Heap，必須等 GC 來收。
		優化後：既然這物件只在方法內有用，JVM 直接把它丟進 Stack。
		好處：方法結束，物件隨 Stack 直接銷毀。完全不需要 GC 參與，效率提升極大！

	B. 標量替換 (Scalar Replacement) —— 拆解對象
		傳統邏輯：這是一個 Point {int x, int y} 物件。
		優化後：JVM 發現你只用了 x 和 y，它乾脆不建立物件了，直接把物件拆成兩個獨立的 int x 和 int y 存在暫存器或棧裡。
		好處：連物件頭（Object Header）的記憶體開銷都省了，這就是你之前學過的「連續記憶體」最快讀取法。

	C. 鎖消除 (Lock Elision) —— 自動解鎖
		傳統邏輯：你在方法裡用了 StringBuffer（內建 synchronized 鎖）。
		優化後：JVM 發現這個物件只有你這個執行緒在用，根本不會有競爭。它會直接把所有的 synchronized 拿掉。
		好處：避開了昂貴的鎖競爭與 CPU 原子操作開銷。

SQL 資料庫
	B+ Tree 扁平型tree(用空間換時間) 來加快查詢速度
		不用二元樹的原因 太高 找一個值 可能就要跑n層 = 從硬碟搬運n次資料
		B+ Tree 1.扁平化 每層有許多索引 3-4層就能裝下所有資料 
				2.連續性 底層用鏈結串起來 所以如果你要找「5月到8月的訂單」，只要找到 5 月，後面直接往後掃就行了(空間局部性)
				
	索引 
		如同書的目錄
		但太多的話 每次更新資料 都要重寫一次B+tree 導致
			1.速度變慢
			2.空間浪費 (因為索引也是資料)
			
	ACID(事務特性)
		Atomicity (原子性)：要嘛全做，要嘛不做 (Undo Log)。
		Consistency (一致性)：符合資料庫完整性約束（如：餘額不為負）。
		Isolation (隔離性)：併發時互不干擾，防止讀到一半的髒資料 (MVCC/Locks)。
		Durability (持久性)：提交後永久保存，不怕斷電 (Redo Log)。
		1.Redo Log (重做日誌)：
			邏輯：當你改資料時，資料庫先「隨手」把動作記在 Redo Log 裡，然後就跟你說「成功了」。
			用途：萬一斷電，重開機時讀取 Redo Log，把沒寫進硬碟的資料「重做」一遍。這就是 持久性 (Durability)。
		2.Undo Log (回滾日誌)：
			用途：如果你事務執行到一半失敗了，或是要讀取「別人還沒改完之前的舊資料」，就靠 Undo Log 把資料「變回去」。這就是 隔離性 (Isolation)。
		恢復機制 (Recovery)
			Redo 第一：重啟後先把能救的（完整的 Redo Log）全部救回來，重建現場。
			Undo 第二：現場重建後，檢查哪些事務「死在一半（沒 Commit）」，利用 Undo Log 把它們清理乾淨。
			損壞處理：如果 Redo Log 紀錄不完整（剪半），則視為無效，直接捨棄，確保資料不會處於「半殘」狀態。
			
	SQL 優化器
		優化器就像導航，它會決定要「走索引」還是「全表掃描」。
		最左前綴 (Leftmost Prefix)：
			如果你建了組合索引 (姓, 名)。
			查「姓=張」：OK，索引有用。
			查「姓=張 且 名=三」：OK，索引有用。
			查「名=三」：失效。因為索引是先按「姓」排的，沒了姓，後面的順序就亂了，優化器只好去翻整本書（全表掃描）。
